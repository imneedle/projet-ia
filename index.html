Prédiction de la qualité de l'air

Jeu de données

Nous utilisons 2 API pour obtenir notre jeu de données:

  • OpenWeatherMap → Air Pollution API :

    Cette API nous donne accès aux données de pollution historique (jusqu'au 27 Novembre 2020) pour une position donnée, heure par heure. Outre l'indice de qualité de l'air de base, l'API renvoie des données sur les gaz polluants tels que le monoxyde de carbone (CO), le monoxyde d'azote (NO), le dioxyde d'azote (NO2), l'ozone (O3), le dioxyde de soufre (SO2), l'ammoniac (NH3) et les particules (PM2,5 et PM10).

  • OpenMeteo → Historical Weather API :

    Cette API donne les données météorologiques historiques (remontant jusqu'en Janvier 1940) pour une position donnée. Nous récoltons sur un durée T :

    • Pour chaque heure : humidité relative à 2m, point de rosée à 2m, moyenne de la couverture nuageuse, vitesse du vent à 80m, direction du vent à 80m, température à 80m.
    • Pour chaque jour : la température maximale et minimale à 2m.

Ces deux jeux de données utilisent une combinaison d'observations de stations météorologiques, d'aéronefs, de bouées, de radars et de satellites pour créer un enregistrement complet des conditions météorologiques passées. Ces ensembles de données sont capables de combler les lacunes en utilisant des modèles mathématiques pour estimer les valeurs de diverses variables météorologiques. En conséquence, les ensembles de données de réanalyse peuvent fournir des informations météorologiques historiques détaillées pour des emplacements qui n'auraient peut-être pas eu de stations météorologiques à proximité, tels que les zones rurales ou l'océan ouvert.

Les modèles pour les données météorologiques historiques utilisent une résolution spatiale de 9 km pour résoudre les détails fins près des côtes ou des terrains montagneux complexes. En général, une résolution spatiale plus élevée signifie que les données sont plus détaillées et représentent plus précisément les conditions météorologiques à des échelles plus petites.

Traitement des données

Pour le traitement de données nous avons pris une architecture de project très modulaire afin de bien séparé la partie "fetching" des données, de la partie "traitement" des données.

Nous avons crée des fetcheurs pour récupérer les données sur la pollution et sur la météo de manière succinte et claire. Une partie du traitement des données est effectuée dans les scripts de récupération. La class de base dans api.py contient des méthodes communes pour traiter les réponses d'API. Les classes spécialisées dans pollution_api.py et weather_api.py implémentent des méthodes spécifiques à chaque type de données.

Enfin nous avons un script local_data.py qui permet simplement de prétraiter les données avant de les fournir aux algorithmes de scikit-learn
